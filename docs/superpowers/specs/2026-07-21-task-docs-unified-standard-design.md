# 任务书统一标准设计

> **历史快照：** 本设计描述任务书迁移前的 `docs/tasks/` 结构；当前任务书位于 `docs/team/tasks/`。
>
> 日期：2026-07-21 | 状态：已批准

## 背景

项目有 8 人 × 6 周 = 48 个任务书文件（`docs/tasks/w1/` ~ `docs/tasks/w6/`）。当前 W1 任务书为"从零上手"风格（含完整源码、半天粒度、算法原理推导、行为约束），W2-W6 为"冲刺看板"风格（日粒度要点、无代码、无验证命令）。详细度相差 3-5 倍，对 8 个刚接触项目的新手来说，W2-W6 过于简略，不知道从哪下手、怎么验证自己做对了。

## 目标

将 48 个任务书统一为同一详细度标准：日粒度任务 + 关键代码片段 + 验证命令 + 具体验收标准。W1 精简，W2-W6 补充。

## 统一模板结构

每个任务书遵循以下固定结构：

### 1. 标题与元信息

文件开头是角色名 + 周次标题，紧跟一行引用块写明周期日期范围和一句话核心目标。

```markdown
# Tech Lead W1 任务书

> 周期：7/20–7/26 | 核心目标：冻结核心接口，路口 1 固定配时 + CA-MP 跑通 3600 步
```

### 2. 本周背景（条件性出现）

只在本周首次涉及新概念或新工具时出现，用 3-5 句话解释"这是什么、为什么这周要用它"。不重复讲之前周已经介绍过的内容。如果本周没有新概念，这个段落直接省略。

示例：W2 首次引入 EWMA 预测时写一段简述，W4 首次涉及 Docker 时写一段简述。

### 3. 每日任务

按 Day 1 到 Day 7 逐日列出，每天 3-5 个明确子任务，用 checkbox 格式。每个子任务具体到"做什么、产出什么"，不写笼统的"完成 X 模块"。

每天结尾附一条"验证"指令：跑什么命令、期望看到什么输出。这是队员自检的唯一依据。

```markdown
### Day 3（7/22）

- [ ] 在 `algorithms/ca_max_pressure.py` 中实现 `step()` 方法的容量归一化压力计算
- [ ] 实现溢出门控逻辑：占用率 > 0.9 时返回强制放行 ControlAction
- [ ] 编写单元测试验证：短车道（capacity=6）排队 5 辆时压力值高于长车道（capacity=20）排队 10 辆

**验证：** `python -m pytest tests/test_ca_mp.py -v` → 3 passed
```

### 4. 关键代码指引

给出本周核心工作的函数签名和 10-30 行关键逻辑片段，让队员知道代码的"形状"。片段后标注完整实现所在的仓库文件路径。不贴完整文件源码。

```python

    ...
```

### 5. 交付物

列出本周需要产出的文件，每个文件写明：文件路径、截止日（Day 几）、验收标准。验收标准具体到可判定的程度——写清楚"包含哪些字段"、"跑什么命令通过"、"输出什么结果算对"。

### 6. 协作对接

用 1-3 句话写清楚本周需要和谁对接什么内容。只写"给谁什么"或"从谁那拿什么"，不写操作细节。

## W1 改动方向（精简）

- 砍掉所有完整源码（如 `core/types.py` 全文 70 行、`ca_max_pressure.py` 130 行），替换为 10-30 行关键片段 + 文件路径引用
- 去掉"上午/下午"的半天拆分，统一为日粒度
- 去掉行为约束段落（如"你是唯一可以修改此文件的人"、"不要帮别人写代码"）
- 去掉逐字逐句的 git 操作教学，保留必要的验证命令
- 算法原理讲解压缩到"本周背景"段落的 3-5 句话，不展开公式推导
- 保留交付物和验收标准

## W2-W6 改动方向（补充）

- 每天的任务从 2-3 条笼统描述扩展为 3-5 条具体子任务
- 每天补充至少一条验证命令
- 补充"关键代码指引"段落
- 交付物验收标准从一句话扩展为具体条件
- 首次出现新概念时补充 3-5 句背景说明

## 执行流程

1. 先改写 TL 的 W1-W6 共 6 个文件作为模板
2. 用户确认风格（重点检查：验证命令能否跑通、代码片段与仓库实际代码是否一致）
3. 批量套用到其余 7 个角色的 42 个文件

## 不做的事

- 不改变各周的任务内容和分工安排，只统一表达格式和详细度
- 不改变任何人的工作范围——W2-W6 "扩展为 3-5 条子任务"是把原有笼统描述拆细，不是加新活
- 不修改 `docs/tasks/roadmap.md` 总路线图
