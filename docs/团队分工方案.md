# 团队分工方案（定稿）

> 比赛：XH-202613 面向雄安新区"城市大脑"的车路云一体化协同管控算法与仿真平台研究
> 团队：8人（大二），4组 × 2人
> 截止日期：2026年9月15日

---

## 总原则

> 现阶段所有工作仅限于**学习、理解、整理、规划**，不进行算法实现、不修改代码、不开始实验，只为后续开发建立统一认知和规范。

> **所有共享资料先进入知识库，再进入正式文档；正式报告、PPT、视频均不作为当前阶段的工作目标。**

---

## 组织结构：4组 × 2人

| 组别 | 定位 | 当前阶段核心任务 |
|------|------|-----------------|
| A 数据与仿真组 | 基础设施层 | 摸清数据、跑通仿真、统一接口认知 |
| B 算法研究组 | 核心智力层 | 调研算法、分析需求、积累候选方案 |
| C 实验评估组 | 验证度量层 | 学习评价指标、设计实验草案、整理数据字段 |
| D 项目管理与文档组 | 组织协调层 | 需求整理、规范制定、资料沉淀、任务跟踪 |

---

## 各组负责人（Owner）制度

| 小组 | Owner职责 |
|------|----------|
| A组 | 组织本组工作、同步进度、对外沟通 |
| B组 | 组织算法调研、汇总结论、对外沟通 |
| C组 | 组织指标讨论、统一实验规范、对外沟通 |
| D组 | 主持周会、维护任务板、汇总周报 |

规则：
- 每组只有 1 名对外负责人（Owner）。
- 组内两人平级，Owner 不代表工作更多，只负责沟通和协调。
- 跨组问题统一由 Owner 对接，避免 8 个人互相沟通导致信息混乱。

---

## A 数据与仿真组（2人）

**贯穿职责：** 仿真环境的建设者与维护者

**当前阶段工作：**
- 逐个打开20个路口的SUMO工程，确认能否正常运行（记录状态：能跑/报错/版本不兼容）
- 整理每个路口的基础信息：进口道数量、车道数、流量大小、转向比例、现状配时方案
- 阅读 `.net.xml` 理解路口拓扑，阅读 `.flow.xml` / `.turn.xml` 理解交通需求
- 打开 `.xlsx`，提取现状信号配时方案
- 查看高精地图 `.png`，对照 net.xml 确认路口几何
- 明确仿真输出数据格式（tripinfo / fcd / summary 各包含什么字段）
- 确认 SUMO 版本兼容性（数据由至少5个版本生成：1.13.0、1.18.0、1.23.1、1.26.0、1.27.1）

**当前阶段产出（共享知识库）：**
- 20路口数据档案：每个路口一份资料卡（拓扑、流量、配时、地图截图、运行状态）
- 20路口数据总览表：关键参数一览（供B/C组快速查阅）
- SUMO输出字段说明：tripinfo.xml / traj.xml / stats.xml 各字段含义

**后续自然过渡到：** 批量仿真脚本、算法接入调试、系统集成测试、部署文档

---

## B 算法研究组（2人）

**贯穿职责：** 算法方案的设计者与技术决策者

**当前阶段工作：**
- 精读比赛PDF中路线B的要求，明确评分要点（经典算法 + 场景适配 + 参数调优 + 高级扩展）
- 文献调研：Webster配时优化、SCOOT/SCATS自适应控制、绿波协调、区域协调控制等
- 了解"窄路密网"场景特征：间距短、交叉口密、转向比例高
- 研究 algorithm.py 的 TraCI 接口能力：能获取什么数据、能控制什么对象、每步能做什么
- 积累2-3个候选算法方向的对比信息（优劣、实现难度、创新性、与评分标准匹配度）
- 关注高级扩展方向：分布式/边缘计算版本、ML预测流量辅助决策

**当前阶段产出（共享知识库）：**
- 算法调研笔记：每篇文献/方法的核心思路、适用场景、优缺点（不急着下最终结论）
- 候选方案对比表：2-3个方向的多维度对比（供后续团队讨论决策）
- TraCI接口能力清单：能读什么、能控什么、调用频率限制

**后续自然过渡到：** 确定算法方案、详细设计、编码实现、参数调优

---

## C 实验评估组（2人）

**贯穿职责：** 实验体系的设计者与质量把关者

**当前阶段工作：**
- 研读评分标准中"实验验证与性能评估"（20分）的具体得分点
- 学习交通仿真常用评价指标：平均延误、排队长度、通行能力、停车次数、行程时间等
- 研究 SUMO 输出文件能提供哪些原始数据（与A组协作确认字段）
- 设计对比实验框架草案：算法 vs 固定配时，单路口 vs 多路口，低/中/高流量
- 规划可视化方案：需要哪些图表类型（对比柱状图、时序曲线、时空轨迹图）
- 确定数据格式规范草案：实验结果以什么格式存储，方便批量处理和绘图

**当前阶段产出（共享知识库）：**
- 评价指标说明：每个指标的定义、计算公式、数据来源字段
- 实验设计草案：实验矩阵（场景×参数×对比方案）、预期图表清单
- SUMO输出数据字段整理：哪些字段对应哪些指标、如何计算

**后续自然过渡到：** 编写数据处理脚本、执行批量实验、生成图表、撰写实验报告

---

## D 项目管理与文档组（2人）

**贯穿职责：** 项目的组织者、规范的制定者、最终文档的整合者

**当前阶段工作：**
- 精读比赛PDF，提取全部交付物要求和评分细则
- 建立项目协作基础设施：Git仓库结构、分支规范、文件命名规范、提交规范
- 制定会议机制：周会频率、会议记录模板、进度汇报格式
- 搭建技术报告框架：目录结构、各章节对应哪组产出
- 搭建PPT模板：确定风格、页面结构
- 构思演示视频脚本大纲：场景选取、叙事线、时长分配
- 跟踪各组产出进度，汇总为项目周报
- 整理比赛相关参考资料、雄安新区规划资料
- 维护共享知识库的目录结构与索引

**当前阶段产出（共享知识库）：**
- 需求与评分要点清单：从PDF提取的全部要求，按交付物和评分维度整理
- 项目规范文档：Git规范、命名规范、会议制度、文件组织结构
- 文档模板：技术报告框架、PPT模板、视频脚本大纲
- 资料索引：共享知识库的目录说明，方便全员查找
- 会议记录与周报

**后续自然过渡到：** 整合技术报告、制作PPT、录制/剪辑视频、组织答辩演练

---

## 组间协作关系（当前阶段）

```
D 项目管理与文档组
  ├─ 向全员发布：需求清单、协作规范、知识库结构
  └─ 从各组收集：阶段产出、进度更新

A 数据与仿真组 ──→ 路口数据总览表 ──→ 供 B/C 查阅
B 算法研究组   ──→ 候选方案对比表 ──→ 供 C 设计实验、D 搭报告框架
C 实验评估组   ──→ 指标与数据字段 ──→ 供 B 确认算法需输出什么
```

四组并行，无硬阻塞。唯一软依赖：B组调研时需要A组提供路口基本特征，A组第一周即可给出。

---

## 机动性规则

- 每组2人互为AB角，任何一人缺席另一人可独立推进
- 当前阶段D组工作量相对轻，可支援A组验证SUMO工程
- 进入实现阶段后，C组可协助B组做参数扫描实验
- 最终冲刺阶段，A/B/C组向D组交付素材，D组主导整合，全员审阅
