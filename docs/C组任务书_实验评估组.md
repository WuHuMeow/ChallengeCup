### 任务2：建立评价指标体系

#### 2.1 学习交通仿真常用评价指标

对以下每个指标，整理其定义、计算公式、物理含义：

**效率类指标：**

| 指标 | 定义 | 计算公式 | 单位 | 越小越好/越大越好 |
|------|------|---------|------|----------------|
| 平均行程时间 | 车辆从进入路网到离开路网的平均耗时 | Σ(每辆车行程时间) / 车辆总数 | 秒 | 越小越好 |
| 平均延误 | 车辆实际行程时间与自由流行程时间的差值 | Σ(实际时间 - 自由流时间) / 车辆总数 | 秒 | 越小越好 |
| 平均速度 | 路网中所有车辆的平均行驶速度 | Σ(每辆车速度) / 车辆数 | m/s | 越大越好 |
| 通行量/吞吐量 | 仿真时段内完成行程的车辆总数 | arrived车辆计数 | 辆 | 越大越好 |
| 平均排队长度 | 各进口道排队车辆数的时间平均值 | Σ(每步排队数) / 步数 | 辆 | 越小越好 |
| 最大排队长度 | 仿真期间出现的最大排队车辆数 | max(每步排队数) | 辆 | 越小越好 |

**安全/稳定性指标：**

| 指标 | 定义 | 计算公式 | 单位 |
|------|------|---------|------|
| 停车次数 | 车辆行程中完全停车的次数 | 从tripinfo的waitSteps或stopTime统计 | 次 |
| 碰撞次数 | 仿真中发生的车辆碰撞事件数 | 从stats.xml的collisions字段累计 | 次 |
| 传送次数 | 车辆被SUMO强制传送的次数（表示严重拥堵） | 从stats.xml的teleports字段累计 | 次 |

**能耗/环境指标（加分项）：**

| 指标 | 定义 | 说明 |
|------|------|------|
| 燃油消耗 | 车辆行程中的燃油消耗量 | SUMO可输出，需要配置emission-output |
| CO2排放 | 车辆行程中的二氧化碳排放量 | SUMO可输出，需要配置emission-output |
| 等待时间 | 车辆处于停车状态的累计时间 | 从tripinfo的waitingTime统计 |

**公平性指标（可选）：**

| 指标 | 定义 | 说明 |
|------|------|------|
| 各方向延误方差 | 不同进口道延误的离散程度 | 反映是否存在某方向被严重忽视 |
| 最大延误 | 所有车辆中延误最大的那辆 | 反映极端情况 |

#### 2.2 确定核心指标集

从上述指标中选出5-8个作为核心评价指标（不需要全用，但要覆盖"效率+安全"两个维度）：

**建议核心指标（供讨论）：**
1. 平均行程时间（效率，最直观）
2. 平均延误（效率，信号控制领域最常用）
3. 平均排队长度（效率，反映拥堵程度）
4. 通行量/吞吐量（效率，反映路网处理能力）
5. 平均速度（效率，直观）
6. 停车次数（安全/舒适）
7. 碰撞次数（安全，必须为0或极低）
8. 传送次数（稳定性，必须为0）

#### 2.3 整理指标与SUMO输出字段的映射关系

**从 stats.xml（summary-output）可直接获取：**

| 指标 | 对应字段 | 计算方法 |
|------|---------|---------|
| 平均速度 | meanSpeed | 直接读取，取所有步的平均 |
| 平均速度相对值 | meanSpeedRelative | 直接读取 |
| 碰撞次数 | collisions | 取最后一步的值（累计值） |
| 传送次数 | teleports | 取最后一步的值（累计值） |
| 停车车辆数 | halting | 取所有步的平均 |
| 平均等待时间 | meanWaitingTime | 取所有步的平均 |
| 平均行程时间 | meanTravelTime | 取所有步中非-1值的平均 |
| 吞吐量 | arrived | 取所有步的累加 |
| 路网车辆数 | running | 取所有步的平均（反映拥堵） |

**从 tripinfo-output 可获取（每辆车一行）：**

| 指标 | 对应字段 | 计算方法 |
|------|---------|---------|
| 行程时间 | duration | 所有车辆取平均 |
| 行程长度 | routeLength | 用于计算自由流时间 |
| 时间损失 | timeLoss | 所有车辆取平均（≈延误） |
| 等待步数 | waitSteps | 所有车辆取平均或累加 |
| 等待时间 | waitingTime | 所有车辆取平均 |
| 出发延误 | departDelay | 所有车辆取平均 |

**从 traj.xml（fcd-output）可获取（每步每辆车一行）：**

| 指标 | 对应字段 | 计算方法 |
|------|---------|---------|
| 瞬时速度 | speed | 按时间/空间聚合 |
| 车辆位置 | x, y | 用于绘制时空轨迹图 |
| 所在车道 | lane | 用于统计各车道/进口道排队 |
| 排队长度 | 由lane+speed<阈值推算 | 统计每步各进口道速度<0.1的车辆数 |

**从 queues.xml（queue-output，仅路口11-13、15-20有此输出）可获取：**

| 指标 | 对应字段 | 计算方法 |
|------|---------|---------|
| 各车道排队长度 | queueLength | 直接读取，按车道/时间聚合 |
| 排队尾部位置 | queueLengthEnd | 用于空间排队分析 |
| 车道ID | lane | 区分不同进口道 |

注意：路口1-10、14的sumocfg中未配置queue-output，无queues.xml数据。对这11个路口，排队长度需从traj.xml（speed<阈值的车辆数）或tripinfo中推算。

**注意：需要与A组确认的字段问题：**
- tripinfo-output的实际输出路径（全部20个路口的sumocfg中均配置为`tripinfo.xml`）
- tripinfo的完整字段列表（需要实际运行一次确认）
- 排队长度是否有更直接的获取方式（如通过traci.lane.getLastStepVehicleNumber）
- queues.xml的完整字段列表（需查看已有文件或实际运行确认）

### 任务3：设计对比实验方案草案

#### 3.1 实验对比方案

| 对比维度 | 方案A（基线） | 方案B（算法） |
|---------|-------------|-------------|
| 信号控制方式 | 固定配时（从xlsx提取的现状方案） | 团队最终确定的算法 |
| 说明 | 代表"无优化"的基准水平 | 代表"有优化"的算法效果 |

#### 3.2 实验矩阵（场景×参数）

**场景维度：**

| 场景类型 | 具体设置 | 目的 |
|---------|---------|------|
| 单路口-低流量 | 选1个路口，流量×0.5 | 验证算法在轻交通下的表现 |
| 单路口-中流量 | 选1个路口，原始流量 | 验证算法在正常交通下的表现 |
| 单路口-高流量 | 选1个路口，流量×1.5或×2.0 | 验证算法在拥堵下的表现 |
| 多路口 | 选3-5个连续路口 | 验证算法的协调效果 |
| 全路网 | 20个路口全部 | 验证算法的普适性 |

**参数维度（待B组确定算法后细化）：**
- 算法关键参数取3-5个不同值
- 例如：如果是自适应控制，可以调整"检测周期"、"最小绿灯时间"、"最大周期"等

**实验矩阵草案：**

| 实验编号 | 路口 | 流量水平 | 控制方案 | 算法参数 | 重复次数 |
|---------|------|---------|---------|---------|---------|
| E01 | 路口1 | 原始 | 固定配时 | - | 3 |
| E02 | 路口1 | 原始 | 算法 | 默认 | 3 |
| E03 | 路口1 | ×0.5 | 固定配时 | - | 3 |
| E04 | 路口1 | ×0.5 | 算法 | 默认 | 3 |
| E05 | 路口1 | ×1.5 | 固定配时 | - | 3 |
| E06 | 路口1 | ×1.5 | 算法 | 默认 | 3 |
| ... | ... | ... | ... | ... | ... |

**注意：**
- 每组实验建议重复3次（SUMO有随机性，取平均值更可靠）
- 流量调整方法：修改flow.xml中的number属性（按比例缩放）
- 具体选哪些路口，等A组提供总览表后根据路口特征选取（选有代表性的）

#### 3.3 可视化方案规划

**需要制作的图表类型：**

| 图表类型 | 内容 | 用途 |
|---------|------|------|
| 对比柱状图 | 算法 vs 固定配时，各指标并列对比 | 直观展示提升效果 |
| 指标时序曲线 | 横轴=仿真时间，纵轴=指标值（如排队长度随时间变化） | 展示动态效果 |
| 流量-效果曲线 | 横轴=流量水平，纵轴=指标值，两条线（算法/基线） | 展示不同流量下的表现 |
| 参数敏感性图 | 横轴=算法参数值，纵轴=指标值 | 展示参数调优效果 |
| 时空轨迹图 | 从traj.xml提取车辆轨迹，绘制时间-空间图 | 展示绿波效果/拥堵消散过程 |
| 多路口热力图 | 横轴=路口编号，纵轴=时间，颜色=排队长度 | 展示区域协调效果 |
| 雷达图/蛛网图 | 多维度指标综合对比 | 展示算法的综合优势 |

**图表规范要求（评分标准提到"图表规范"）：**
- 所有图表必须有标题、坐标轴标签、单位、图例
- 对比图中算法和基线用不同颜色/线型区分
- 数据保留合理精度（时间精确到秒，速度精确到0.1m/s）
- 图表编号连续，在报告中可引用

#### 3.4 数据格式规范草案

**实验结果存储格式（建议）：**

每次实验运行后，将结果整理为一行记录：

| 字段 | 说明 | 示例 |
|------|------|------|
| exp_id | 实验编号 | E01 |
| intersection | 路口编号 | 1 |
| flow_level | 流量水平 | original / x0.5 / x1.5 |
| control_method | 控制方案 | fixed / algorithm |
| algorithm_param | 算法参数 | default / param1 / param2 |
| run_id | 重复次数 | 1 / 2 / 3 |
| avg_travel_time | 平均行程时间(s) | 45.2 |
| avg_delay | 平均延误(s) | 12.3 |
| avg_queue | 平均排队长度(辆) | 3.5 |
| throughput | 通行量(辆) | 1650 |
| avg_speed | 平均速度(m/s) | 11.2 |
| stop_count | 平均停车次数 | 1.8 |
| collisions | 碰撞次数 | 0 |
| teleports | 传送次数 | 0 |

**存储格式建议：** CSV或Excel，方便后续用Python/R绑图

## 需要学习的知识

### 必学（开始工作前）

1. **交通评价指标体系**
   - 什么是延误（控制延误、行程延误、停车延误的区别）
   - 什么是服务水平（LOS，A-F六级）
   - 什么是排队长度（时间排队、空间排队）
   - 什么是通行能力（饱和流率）
   - 推荐入门：《交通工程学》或《交通信号控制》教材中的"交通流参数"章节
   - 推荐搜索："traffic signal performance measures"、"intersection level of service"

2. **SUMO输出数据解读**
   - summary-output（stats.xml）：每仿真步的全局统计
   - fcd-output（traj.xml）：每步每辆车的轨迹数据
   - tripinfo-output：每辆车完成行程后的汇总数据
   - 推荐阅读：
     - https://sumo.dlr.de/docs/Simulation/Output/Summary.html
     - https://sumo.dlr.de/docs/Simulation/Output/FCDOutput.html
     - https://sumo.dlr.de/docs/Simulation/Output/TripInfo.html

3. **对比实验设计基础**
   - 什么是对照实验（控制变量法）
   - 为什么需要重复实验（随机性）
   - 什么是基线/基准（baseline）
   - 什么是参数敏感性分析
   - 推荐搜索："experimental design traffic simulation"、"SUMO experiment setup"

### 选学（有余力时）

4. **数据可视化最佳实践**
   - Matplotlib绑图规范（标题、标签、图例、配色）
   - 交通领域常用图表类型
   - 推荐搜索："traffic data visualization"、"matplotlib best practices"

5. **统计分析基础**
   - 均值、标准差、置信区间
   - 如何判断两组数据是否有显著差异（t检验）
   - 这在"量化算法提升效果"时会用到

6. **交通仿真中的随机性**
   - SUMO的随机种子（random seed）
   - 为什么同一配置跑多次结果不同
   - 如何处理随机性（多次运行取平均）

- **不修改任何文件**：不调整XML配置、不修改流量文件
- **不做算法评价**：不判断"哪个算法好"（那是B组调研+团队讨论的事）
- **不写正式实验报告**：产出是共享知识库草案，不是最终实验评估报告
- **不制作正式图表**：只规划"需要什么图"，不实际绑制（没有数据可画）
- **不制作视频**：视频由D组负责，C组只提供数据需求清单
- **不确定最终实验方案**：草案是"供讨论的框架"，算法确定后还需细化

## 后续将自然过渡到哪些工作（实现阶段）

当团队确定算法方案、进入开发阶段后，C组的工作将自然过渡为：

1. **编写数据处理脚本**：解析stats.xml、traj.xml、tripinfo，计算各指标
2. **编写批量实验脚本**：按实验矩阵自动运行多组仿真（与A组协作）
3. **执行实验**：按方案跑完所有实验组合，收集原始数据
4. **数据清洗与整理**：处理异常值、计算统计量（均值、标准差）
5. **制作可视化图表**：按可视化方案绑制所有图表
6. **撰写实验评估报告**：数据分析、结论、算法提升效果量化
7. **为演示视频提供数据素材**：关键帧数据、实时指标数据、对比数据
8. **配合B组调参**：执行参数敏感性实验，反馈结果给B组
