### 任务2：了解"窄路密网"场景特征

雄安新区的路网特征与传统城市不同，需要理解这些特征对算法设计的影响：

**需要调研和整理的特征：**
- 路网密度高：交叉口间距短（可能200-400米），信号控制需要协调
- 道路等级低：以次干路和支路为主，车道数少（从数据看每方向2-3车道）
- 转向比例高：短间距导致车辆频繁转向，直行比例相对较低
- 流量特征：各路口流量差异极大。路口1约366-462辆/小时/方向；路口18高达834辆/小时；路口20最低仅183辆/小时。部分路口只有3个方向流，部分有5个。需参考A组总览表了解全貌
- 混合交通：可能有非机动车和行人干扰（从比赛背景描述推断）

**这些特征对算法的约束：**
- 单路口优化不够，需要多路口协调
- 周期不宜过长（短间距下车辆很快到达下一个路口）
- 绿波协调的价值大（短间距容易形成绿波带）
- 实时性要求高（流量变化快，固定配时效果差）

**信息来源：**
- 比赛PDF中的背景描述
- A组提供的20路口数据总览表（等A组完成后查阅）
- 雄安新区公开规划资料（D组会整理到知识库）
- 文献中关于"窄路密网"或"高密度路网"信号控制的研究

### 任务3：文献调研——经典交通管控算法

对以下每个方向，阅读至少2-3篇文献或教程，整理笔记：

#### 方向一：Webster配时优化及其改进

- 核心思路：基于路口各方向流量，用Webster公式计算最优周期和绿信比
- 公式：C₀ = (1.5L + 5) / (1 - Y)，其中L为总损失时间，Y为总流量比
- 适用场景：单路口、稳态流量
- 优点：简单、有解析解、易于实现
- 缺点：假设流量稳定，无法应对实时变化，不考虑路口间协调
- 改进方向：动态Webster（用实时检测流量替代历史流量）、多时段Webster
- 与比赛的匹配度：实现简单，但创新性有限，适合作为基线对比

#### 方向二：SCOOT/SCATS类自适应控制

- SCOOT核心思路：实时检测交通流，用预测模型（如CYCLIC）预测未来流量，在线优化绿信比、相位差、周期
- SCATS核心思路：基于饱和度（degree of saturation）实时调整绿信比，子区协调
- 适用场景：多路口、流量波动大
- 优点：实时响应、多路口协调、工程验证充分
- 缺点：实现复杂、需要检测器数据、原始算法有专利限制
- 改进方向：简化版SCOOT（只用排队长度做反馈）、结合ML预测替代CYCLIC模型
- 与比赛的匹配度：高——体现"自适应"和"协同"，创新性空间大

#### 方向三：绿波协调控制（MAXBAND/MULTIBAND）

- 核心思路：在一条干线上，通过优化各路口相位差，使车辆以建议速度行驶时连续遇到绿灯
- MAXBAND：最大化绿波带宽（单向或双向）
- MULTIBAND：考虑多时段、多方向的绿波协调
- 适用场景：干线协调、间距较均匀的路口序列
- 优点：效果直观（通行效率提升明显）、数学模型成熟
- 缺点：对路网形态有要求（需要连续路口序列）、对流量变化敏感
- 改进方向：动态绿波（根据实时流量调整相位差）、区域绿波（网格路网）
- 与比赛的匹配度：中高——"窄路密网"天然适合协调控制，但需要确认20个路口是否构成可协调的序列

#### 方向四（备选）：基于强化学习的信号控制

- 核心思路：将信号控制建模为马尔可夫决策过程，用DQN/PPO等算法训练智能体
- 适用场景：复杂路口、难以用规则描述的场景
- 优点：理论上能找到最优策略、创新性强
- 缺点：训练时间长、可解释性差、对仿真环境依赖大、实现难度高
- 与比赛的匹配度：中——路线B强调"经典算法"，纯RL可能偏题；但"AI与传统控制结合"是评分标准明确提到的创新方向
- 注意：如果选这个方向，建议作为"高级扩展"而非主体算法

### 任务4：研究TraCI接口能力

TraCI（Traffic Control Interface）是SUMO提供的实时控制接口，算法通过它在每个仿真步获取数据、下发控制指令。

**需要搞清楚的问题：**

1. **能读取什么数据（输入）：**
   - 车辆信息：`traci.vehicle.getIDList()`、`traci.vehicle.getSpeed(vehID)`、`traci.vehicle.getPosition(vehID)`、`traci.vehicle.getLaneID(vehID)`
   - 车道信息：`traci.lane.getLastStepVehicleNumber(laneID)`（排队车辆数）、`traci.lane.getLastStepMeanSpeed(laneID)`
   - 检测器信息：`traci.inductionloop.getLastStepVehicleNumber(detID)`
   - 信号灯信息：`traci.trafficlight.getRedYellowGreenState(tlsID)`（当前灯色）
   - 仿真信息：`traci.simulation.getTime()`

2. **能控制什么对象（输出）：**
   - 信号灯相位：`traci.trafficlight.setRedYellowGreenState(tlsID, state)`（直接设置灯色）
   - 信号灯相位切换：`traci.trafficlight.setPhase(tlsID, phaseIndex)`（切换到指定相位）
   - 信号灯程序：`traci.trafficlight.setProgramLogic(tlsID, logic)`（修改整个配时方案）
   - 车辆速度建议：`traci.vehicle.setSpeed(vehID, speed)`（如果做车速引导）
   - 车辆路由：`traci.vehicle.setRoute(vehID, edgeList)`（如果做路径引导）

3. **调用约束：**
   - 每个仿真步只能调用一次 `traci.simulationStep()`（注意：路口1-10、14步长为1秒，3600步=3600秒；路口11-13、15-20步长为0.1秒，3600步=360秒）
   - 算法逻辑必须在 `traci.simulationStep()` 之前执行
   - 控制指令在当前步结束时生效
   - 从algorithm.py模板看，循环3600步。步长1秒的路口（1-10、14）仿真3600秒=1小时；步长0.1秒的路口（11-13、15-20）仿真360秒=6分钟

4. **从algorithm.py模板理解算法框架：**
   注意：实际模板中 `traci.start` 的参数是占位符 `"你的文件名.sumocfg"`，使用时需替换为实际文件名如 `"demo_1.sumocfg"`。
   ```python
   traci.start([sumoBinary, "-c", "demo_N.sumocfg"])  # 模板中为"你的文件名.sumocfg"
   for step in range(3600):
       # 1. 获取当前交通状态（车辆数、排队长度、速度等）
       # 2. 根据算法逻辑做决策（切换相位/调整绿信比/发出速度建议）
       # 3. 下发控制指令
       traci.simulationStep()  # 推进一步仿真
   traci.close()
   ```

**需要阅读的TraCI文档：**
- TraCI主文档：https://sumo.dlr.de/docs/TraCI.html
- 信号灯控制接口：https://sumo.dlr.de/docs/TraCI/Change_Traffic_Lights_State.html
